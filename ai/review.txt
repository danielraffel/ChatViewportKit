Here’s the spec I’d hand an agent.

Recommendation

Do not build this on top of ScrollView + LazyVStack as the core engine. Keep the authoring surface SwiftUI-native, but make the runtime engine a UICollectionView with a custom vertical chat layout, hosted in SwiftUI through one UIViewControllerRepresentable. Use SwiftUI views inside cells via UIHostingConfiguration, use a diffable data source for updates, and use a custom UICollectionViewLayout so the engine can directly control contentOffsetAdjustment, targetContentOffset, and insertion animation attributes. That gives you the low-level hooks SwiftUI’s scroll APIs don’t expose while still letting app teams write rows in SwiftUI. It also aligns with Apple’s modern collection-view stack and works with the newer scroll-edge / Liquid Glass behavior instead of fighting it.  ￼

Why this is hard

This is hard because chat is not really a “scroll view problem.” It is a viewport invariants problem. You are trying to preserve one of several mutually competing truths while the system is re-laying out: “bottom stays pinned,” “the currently visible history doesn’t jump,” “the first few messages start at the bottom when content underfills,” “scroll-to-id lands exactly,” and “new cells animate from the bottom.” SwiftUI now has APIs for default anchoring and semantic scroll position — defaultScrollAnchor, scrollPosition, and ScrollViewReader/scrollTo — but those are still higher-level than the pixel-precise offset correction you need when cells self-size, mutate height after async work, or when keyboard and safe-area geometry change mid-update. UICollectionView’s layout invalidation and target-offset hooks are the right layer for that.  ￼

The keyboard and top blur make it worse. Apple’s keyboard architecture became more asynchronous with the out-of-process keyboard on iPhone, and Apple’s guidance is to use keyboardLayoutGuide rather than hand-rolled keyboard frame math. On the visual side, modern iOS scroll-edge effects and the newer Liquid Glass system expect a real, upright scroll surface moving beneath bars; that is exactly why inversion hacks are so fragile here.  ￼

Product decision

Build a framework called ChatViewportKit.

The important decision is this: SwiftUI on the outside, UICollectionView on the inside. Not pure SwiftUI. Not UIScrollView with a hosted lazy stack. Not inversion. Not a one-off message screen. The framework should expose a chat-friendly SwiftUI component, but its engine should be a dedicated collection-view controller. UICollectionView is the better base than UITableView here because Apple’s list stack for collection views is explicitly positioned as a replacement for table-style lists with more layout flexibility, and UICollectionView gives you the layout-level hooks you need for exact offset control and custom insertion animation.  ￼

I would target iOS 16+ for v1. That keeps the integration clean because UIHostingConfiguration is available there, and Apple also added self-resizing behavior for SwiftUI-hosted collection/table cells in that generation. The framework should be optimized for iOS 26 scroll-edge behavior, but not require it.  ￼

The external API should feel like a better ScrollViewReader: rows are authored in SwiftUI, IDs are stable, and imperative scroll commands live in a controller object instead of leaking UIKit into app code.

Internal architecture

1) ChatViewportViewController

A UIKit controller owns the collection view, keyboard coordination, safe-area behavior, and command queue. SwiftUI embeds it with UIViewControllerRepresentable. This keeps keyboard timing, navigation-bar interaction, and scroll command sequencing at the view-controller layer where UIKit expects them.  ￼

2) ChatCollectionView

Use a plain UICollectionView with one section, vertical scrolling, diffable data source, and cell registrations. Never use reloadData() for normal updates. Apply diffs, and use reconfigureItems for content changes that should preserve existing visible cells.  ￼

3) ChatLayout

This is the core differentiator.

Use a custom UICollectionViewLayout specialized for a one-dimensional, bottom-aware chat transcript. It owns:
	•	height cache keyed by item ID + width bucket
	•	prefix-sum index for fast Y-position lookup
	•	underflow bottom anchoring
	•	visible-range lookup
	•	update-time offset correction
	•	custom insertion attributes

Apple’s layout system gives you the exact hooks needed here: prepare(forCollectionViewUpdates:), initialLayoutAttributesForAppearingItem, finalLayoutAttributesForDisappearingItem, targetContentOffset, and invalidation contexts with contentOffsetAdjustment.  ￼

4) CellHost

Each message cell uses UIHostingConfiguration so app teams still build row UI in SwiftUI. Because Apple added self-resizing support for SwiftUI-hosted cells, height changes from text expansion, async image loads, or read-state UI changes can flow through the collection view instead of forcing custom hosting-controller hacks.  ￼

5) KeyboardCoordinator

Use keyboardLayoutGuide to position the composer / bottom accessory and treat keyboard movement as a viewport-height change, not as an ad hoc content inset patch. Let contentInsetAdjustmentBehavior remain compatible with bars and safe areas; only add the framework’s own bottom inset on top of the adjusted inset.  ￼

6) AccessibilityCoordinator

When programmatic scrolls or large transcript updates move focus meaningfully, post .layoutChanged or .pageScrolled notifications as appropriate. The framework should not require users to build custom “screen reader jump-to-id” machinery.  ￼

Core behavioral spec

Requirement 1: first messages must start bottom-anchored

Do not fake this with an initial scrollToBottom.
The layout itself should compute:

underflowTopPadding = max(0, viewportHeight - contentHeight)

and offset all item frames downward by that amount. That means a 1-message transcript, 3-message transcript, and 20-message transcript all naturally settle at the bottom without any startup jump.

Requirement 2: must scroll to bottom, top, or any ID

Maintain an ID → index map in the data source and a height index in the layout. scrollToBottom scrolls to the last item with a bottom anchor. scrollToTop scrolls to the first item. scrollTo(id:) first computes an estimated target rect from cached heights, then refines after layout if needed. This is more reliable than geometry-reader probing because the engine owns the item/frame map.

Requirement 3: no inversion

Hard rule: no transforms on the scroll view, no reversed geometry, no upside-down cell tree. That avoids broken scroll-edge effects, weird accessibility ordering, reversed hit-testing assumptions, and the content-offset sign confusion that makes resize jumps so annoying.

Requirement 4: smooth append animation from bottom even when content underfills

This is where a custom layout matters. For a tail insertion, use initialLayoutAttributesForAppearingItem to place the new cell slightly below its final frame with reduced alpha, then animate it into place. Because bottom anchoring is a layout property, this works whether the transcript is shorter or taller than the viewport.  ￼

Requirement 5: prepending older messages must not jump

Capture a top anchor snapshot before the prepend:
	•	item ID of the first visible message
	•	offset from the viewport top to that item’s frame

After the diff is applied and new frames are computed, use contentOffsetAdjustment to preserve that anchor’s screen position exactly. The key insight is: do not predict heights perfectly; correct the offset after actual layout. That is the clean fix.  ￼

Requirement 6: height changes of existing cells must not wreck history position

Treat height mutations the same way as prepends: capture anchor, relayout, correct offset. If the user is pinned to bottom, preserve bottom distance instead. If the user is reading history, preserve the top anchor. This turns async image loads, message editing, and dynamic type changes into one unified mechanism.

Requirement 7: must work with NavigationStack and the top blur effect

The representable should expose a real upright scroll surface with automatic safe-area participation. Do not disable system scroll-edge behavior unless the host app explicitly opts out. On newer systems, scroll views and toolbars participate in the automatic scroll-edge effect; on earlier systems, the nav bar’s scrollEdgeAppearance should remain compatible.  ￼

Requirement 8: reusable framework, not a one-shot hack

The framework must be transcript-agnostic. Messages are just items with stable IDs and SwiftUI-rendered rows. Date separators, unread markers, typing indicators, load-older spinners, and “jump to latest” affordances should all be modeled as items or supplementary elements, not special-case view code.

State machine

This is the part most implementations miss.

The engine needs an explicit scroll mode:

enum ViewportMode {
    case initialBottomAnchored
    case pinnedToBottom
    case freeBrowsing(anchor: AnchorSnapshot)
    case programmaticScroll(target: ScrollTarget)
    case batchUpdating(policy: AnchorPolicy)
}

And it needs one update pipeline:
	1.	Classify update: append / prepend / mutate / reset / keyboard / bounds change
	2.	Capture anchor snapshot based on current mode
	3.	Apply snapshot diff
	4.	Let layout compute new geometry
	5.	Apply contentOffsetAdjustment
	6.	Optionally run insertion animation
	7.	Recompute pinned/free mode
	8.	Emit callbacks

That single pipeline should handle new messages, loading history, keyboard show/hide, composer growth, rotation, and dynamic type.

What not to do

Do not build v1 around literal LazyVStack.
Do not use inverse scrolling.
Do not measure every row with nested GeometryReaders.
Do not depend on “wait one more runloop” hacks as a correctness strategy.
Do not hand-roll keyboard notifications when keyboardLayoutGuide solves the geometry problem more cleanly.  ￼

Phased delivery plan

Phase 1 — UIKit core spike

Build the collection view, diffable data source, custom layout, and anchor engine with simple UIKit text cells only.
Exit criteria: append, prepend, scrollToBottom, scrollToTop, scrollTo(id), no jumps.

Phase 2 — SwiftUI hosting

Swap cells to UIHostingConfiguration, expose ChatViewport in SwiftUI, and stabilize self-resizing behavior.
Exit criteria: app teams can author message rows in SwiftUI; async height changes preserve position.  ￼

Phase 3 — keyboard + composer + nav integration

Add keyboardLayoutGuide, composer growth handling, scroll indicators, and NavigationStack showcase.
Exit criteria: bottom pin survives keyboard show/hide and multiline composer growth; top blur works.  ￼

Phase 4 — performance and accessibility

Add prefetch hooks, VoiceOver notifications, dynamic type, RTL, and stress tests with large transcripts. UICollectionView supports data prefetching, so the framework should expose a clean hook for older-message/image warmup.  ￼

Phase 5 — packaging and docs

Ship as Swift Package with:
	•	ChatViewportKitCore
	•	ChatViewportKitSwiftUI
	•	ChatViewportKitDemo

The example app should be as important as the framework.

Build a demo called “Transcript Lab.” It should have toggles for:
	•	1, 5, 50, 5,000 messages
	•	append one message / append burst
	•	prepend older history
	•	async image load causing height expansion
	•	edit a message to 5x height
	•	jump to latest
	•	jump to arbitrary ID
	•	dynamic type sizes
	•	keyboard + multiline composer
	•	date separators / unread marker / typing indicator
	•	NavigationStack toolbar demo showing top blur behavior

Also include an on-screen debug HUD:
	•	current mode (pinned, free, programmatic)
	•	distance from bottom
	•	visible item IDs
	•	anchor snapshot currently being preserved
	•	layout invalidation reason

That demo is what proves the framework is real.

Definition of done

The framework is done when these invariants hold:
	•	No visible jump on prepend.
	•	No visible jump on async cell height change.
	•	New tail messages animate from the bottom in both underfilled and overfilled states.
	•	scrollToBottom, scrollToTop, and scrollTo(id:) are deterministic.
	•	Bottom pin survives keyboard show/hide and composer growth.
	•	Navigation blur / scroll-edge behavior remains intact.
	•	Rows are authored in SwiftUI.
	•	Host app integration is one view + one controller object.

The short version is: don’t try to outsmart SwiftUI’s scroll view. Replace the engine, not the row authoring model. The winning design is UICollectionView + custom layout + diffable + UIHostingConfiguration + one representable. That is the clean path to something that actually feels native instead of barely compiling.
